2025 年 11 月 2 日

我频繁使用 Claude Code。

作为一名业余开发者，我每周都会在虚拟机中用它做好几次副业项目，还经常加上`--dangerously-skip-permissions`参数，随心敲代码实现脑海里的各种想法。在工作中，我所在的团队为公司工程团队搭建 AI 集成开发环境的规则和工具，仅代码生成这一项，每月就会消耗数十亿个令牌。

命令行界面Agent领域的竞争日趋激烈，在 Claude Code、Gemini 命令行工具、Cursor 和 Codex 命令行工具之间，这场竞赛的核心参与者其实是 Anthropic 和 OpenAI 两家公司。但说实话，和其他开发者交流后我发现，大家选择工具的依据往往是一些表面因素 —— 比如某个用起来 “顺手” 的功能，或是自己偏爱的系统提示风格。如今这类工具的性能都已相当出色，而且我发现人们也常常过度在意工具的输出格式或用户界面。对我而言，工具那种满口 “你说得太对了！” 的奉承式回应根本不算明显缺陷，反而说明你对工具的使用已经足够深入。我的核心使用原则向来是 “一次性交办，无需后续操心”—— 把任务委托给工具、设定好上下文，就让它自主运行。评判工具的标准，永远是最终的代码合并请求成果，而非它的实现过程。

过去几个月我一直坚持使用 Claude Code，这篇文章就是我对其整个生态系统的使用心得。我会介绍自己用到的几乎所有功能（同样重要的是，也会说说那些我不用的功能），从基础的`CLAUDE.md`文件、自定义斜杠命令，到强大的子Agent、钩子函数和 GitHub Actions。本文篇幅稍长，更适合当作参考手册，而非逐字通读的文章。

## CLAUDE.md 文件

想要高效使用 Claude Code，代码库中最重要的文件就是根目录下的`CLAUDE.md`。这份文件是Agent的 “行为准则”，也是它了解当前代码仓库具体运行规则的核心依据。

如何维护这份文件，取决于实际使用场景：

- 做副业项目时，我会让 Claude 随意在里面写入内容；
- 工作中，我们的单体仓库里的`CLAUDE.md`有严格的维护规范，目前文件大小已达 13KB（我预计很容易就能涨到 25KB）。

这份文件只记录 30% 及以上工程师都会用到的工具和接口（其他工具的文档，会放在对应产品或库的专属 Markdown 文件中）。

我们甚至开始为每个内部工具的文档设定令牌使用上限，几乎相当于给各个团队 “售卖广告位”—— 如果某个工具的用法无法被简洁说明，那它就还没资格被写入`CLAUDE.md`。

### 编写技巧与常见误区

久而久之，我们总结出了一套鲜明且有明确立场的`CLAUDE.md`编写原则：

1. 先定约束规则，而非写使用手册：`CLAUDE.md`的初始内容要简洁，只记录 Claude 频繁出错的场景即可。
2. 避免用 @引用文件文档：如果其他地方有详细文档，很容易想在`CLAUDE.md`中用 @引用这些文件，但这会在每次运行时嵌入整个文件内容，导致上下文窗口臃肿。可如果只写文件路径，Claude 又往往会忽略。正确的做法是，告诉Agent**为何以及何时**需要阅读该文件，比如：“如需复杂的 XX 用法，或遇到`FooBarError`错误时，可参考`path/to/docs.md`中的高级故障排除步骤。”
3. 别只说 “禁止”：避免只用否定式约束，比如 “切勿使用`--foo-bar`参数”。当Agent认为必须使用该参数时，这种表述会让它陷入卡顿。一定要同时给出替代方案。
4. 把`CLAUDE.md`当作优化推手：如果你的命令行指令复杂又冗长，不要写大段文档去解释 —— 这只是在解决人为造成的问题。正确的做法是，编写一个简单的 bash 包装脚本，设计清晰、直观的接口，再将这个脚本记录在文档中。让`CLAUDE.md`保持简洁，能倒逼你简化整个代码库和内部工具。

以下是一份简化的文件示例：


```
# 单体仓库
## Python开发规范
- 始终遵循……
- 使用以下命令测试：<具体命令>
……还有10条规范……

## 内部命令行工具XX
……10条要点，聚焦80%的常用场景……
- <使用示例>
- 始终……
- 避免使用X方式，优先选择Y方式

如需XX复杂用法，或遇到XX错误，请参阅path/to/<tool>_docs.md
……
```

最后，我们会让这份文件与`AGENTS.md`保持同步，确保和工程师可能使用的其他 AI 集成开发环境兼容。

### 核心要点

将`CLAUDE.md`视为一套精心整理的高层级约束规则和文件指引。借助它判断需要在哪些地方打造更适配 AI（也更适配人类）的工具，而非试图把它写成一本详尽的使用手册。

## 精简、上下文与清晰化

我建议在编码过程中，至少运行一次`/context`命令，了解自己如何使用 20 万个令牌的上下文窗口（即便有了 Sonnet-1M 模型，我也不相信整个上下文窗口能被有效利用）。对我们而言，在单体仓库中开启一个新会话，基础令牌消耗就有约 2 万个（占 10%），剩余 18 万个令牌用于代码修改 —— 这部分很容易被快速用完。

这是我近期一个副业项目中`/context`命令的运行截图。你可以把上下文窗口理解成磁盘空间，开发功能的过程中会不断被占用。工作数分钟或数小时后，你需要清理对话消息（紫色部分）来释放空间，才能继续开发。

我主要有三种上下文管理工作流：

1. `compact`（精简，建议避免使用）：我会尽可能不用这个命令。自动精简功能的逻辑不透明、容易出错，且优化效果不佳。
2. `clear` + `catchup`（简单重启，我的默认方式）：先执行`/clear`清空会话状态，再运行自定义的`/catchup`命令，让 Claude 读取当前 git 分支中所有修改过的文件。
3. “记录并清空”（复杂重启，适用于大型任务）：先让 Claude 将开发计划和进度写入一个 md 文件，执行`/clear`清空状态，再开启新会话，让 Claude 读取该 md 文件并继续开发。

### 核心要点

不要相信自动精简功能。简单重启用`/clear`命令，处理复杂任务时，用 “记录并清空” 的方式为其创建持久化的外部 “记忆”。

## 自定义斜杠命令

在我看来，斜杠命令只是常用提示词的简单快捷方式，仅此而已。我的配置十分精简：

- `/catchup`：前文提到的命令，作用是让 Claude 读取当前 git 分支中所有修改过的文件。
- `/pr`：一个简易辅助命令，用于清理代码、暂存修改并准备代码合并请求。

在我看来，如果配置了一长串复杂的自定义斜杠命令，就陷入了使用误区。像 Claude 这样的Agent，核心价值本就是你几乎可以输入任意指令，都能得到可用、可合并的结果。如果非要让工程师（甚至非技术人员）去学习一整套需要专门查阅文档的 “魔法命令” 才能开展工作，那就是工具使用的失败。

### 核心要点

将斜杠命令当作简单的个人快捷方式，而非替代`CLAUDE.md`的优化和工具的完善。

## 自定义子Agent

从理论上看，自定义子Agent是 Claude Code 中最强大的上下文管理功能。其设计思路很简单：一项复杂任务需要 X 个令牌的输入上下文（比如测试运行方法），会累积 Y 个令牌的工作上下文，最终生成 Z 个令牌的结果。执行 N 个任务，主上下文窗口就会占用 (X+Y+Z)×N 个令牌。

子Agent的解决方案，是将 (X+Y)×N 的工作负载分配给专业的子Agent，这些子Agent只向主Agent返回最终的 Z 个令牌结果，从而保持主上下文窗口的整洁。

但实际使用中我发现，这个看似强大的功能会带来两个新问题：

1. 上下文隔离：如果我创建了一个`PythonTests`子Agent，所有与测试相关的上下文就会被隐藏，主Agent无法再从整体上对代码修改进行逻辑推理，甚至必须调用子Agent，才知道如何验证自己编写的代码。
2. 固化工作流：更糟的是，这会把 Claude 限制在人类定义的僵化工作流中。原本我们希望Agent自主解决任务分配问题，结果反而变成了由人类指定它该如何委派工作。

我更倾向的替代方案，是使用 Claude 内置的`Task(...)`功能，生成通用Agent的克隆体。

我会将所有核心上下文写入`CLAUDE.md`，然后让主Agent自主决定何时、如何将工作委派给自身的克隆体。这种方式既能获得子Agent的上下文节省优势，又能规避其弊端，让Agent动态管理工作编排。

在我的《构建多Agent系统（第二部分）》一文中，我将这种架构称为 “主 - 克隆” 架构，并且强烈推荐，而非自定义子Agent所推崇的 “主导 - 专业” 架构。

### 核心要点

自定义子Agent是一种脆弱的解决方案。把上下文都写入`CLAUDE.md`，让主Agent自行使用`Task/Explore(...)`功能管理工作委派即可。

## 恢复、继续与历史记录

基础使用层面，我经常用到`claude --resume`和`claude --continue`命令。它们非常适合重启卡死的终端，或快速恢复之前的会话。我还常会恢复几天前的会话，让Agent总结当时是如何解决某个具体错误的，再根据这些总结优化我们的`CLAUDE.md`和内部工具。

深入使用的话，Claude Code 会将所有会话历史存储在`~/.claude/projects/`目录下，方便调取原始的历史会话数据。我编写了脚本对这些日志进行元分析，找出常见的异常、权限请求和错误模式，从而优化面向Agent的上下文配置。

### 核心要点

使用`claude --resume`和`claude --continue`重启会话，挖掘被隐藏的历史上下文信息。

## 钩子函数

钩子函数的作用至关重要。我做副业项目时不会用，但在复杂的企业级代码仓库中，它是引导 Claude 按规范工作的关键。钩子函数是确定性的 “必须执行” 规则，能与`CLAUDE.md`中 “建议执行” 的规范形成互补。

我们主要使用两种钩子函数：

1. 提交时拦截钩子：这是我们的核心用法。我们配置了一个`PreToolUse`钩子，包裹所有`Bash(git commit)`命令。该钩子会检查是否存在`/tmp/agent-pre-commit-pass`文件 —— 这个文件只有当所有测试通过后，由测试脚本生成。如果文件不存在，钩子会拦截此次提交，迫使 Claude 进入 “测试 - 修复” 循环，直到构建成功。
2. 提示钩子：一种简单的非拦截式钩子，当Agent的操作并非最优时，会向其发送 “一次性” 的反馈提示。

我们刻意不使用 “写入时拦截” 钩子（比如在文件编辑或写入操作时触发）。在Agent执行计划的过程中进行拦截，会让它陷入困惑，甚至出现 “执行中断” 的情况。更有效的方式是，让Agent完成全部工作后，在提交阶段检查最终的完整结果。

### 核心要点

使用钩子函数在提交阶段执行状态验证（提交时拦截）。避免在写入阶段拦截 —— 让Agent完成执行计划后，再检查最终结果。

## 规划模式

对于借助 AI 集成开发环境开发的大型功能，规划模式是必不可少的。

做副业项目时，我只使用内置的规划模式。它能让我在 Claude 开始编码前与其对齐开发思路，既明确功能的实现方式，也设定好 “检查节点”—— 让 Claude 在这些节点暂停工作，向我展示进度。经常使用这个模式，能培养出一种直觉：如何用最少的上下文，让 Claude 制定出合理的开发计划，同时避免其在实现过程中出错。

在工作的单体仓库中，我们开始基于 Claude Code 开发工具包，搭建自定义的规划工具。它和原生规划模式类似，但通过深度提示，让输出内容与公司现有的技术设计格式对齐，还能从一开始就强制遵循公司内部的最佳实践 —— 从代码结构到数据隐私和安全规范。这样一来，工程师就能像资深架构师一样，随心规划新功能的开发思路（至少设计初衷是如此）。

### 核心要点

开发复杂功能时，一定要使用内置的规划模式，在Agent开始工作前对齐开发计划。

## SKILL

我认同西蒙・威利森的观点：SKILL的重要性，或许超过了MCP。

如果你看过我之前的文章，就会知道我在大多数开发工作流中，已经不再使用MCP，反而更倾向于编写简单的命令行工具（我在《AI 读不懂你的文档》一文中也论证过这一点）。我对Agent自主能力的理解，逐渐演变成三个阶段：

1. 单提示词阶段：将所有上下文一次性写入一个超长提示词中（脆弱、无法扩展）。
2. 工具调用阶段：经典的Agent模式，由人类手工打造工具，为Agent抽象现实场景（比上一阶段好，但会产生新的抽象层和上下文瓶颈）。
3. 脚本编写阶段：让Agent直接访问原始的运行环境 —— 包括二进制文件、脚本和文档，由它实时编写代码与环境交互。

基于这个模型，Agent的SKILL显然是下一代核心特性，它是 “脚本编写” 阶段的正式产品化落地。

如果你和我一样，早已偏爱命令行工具而非MCP，那其实一直都在间接享受SKILL的优势。而`SKILL.md`文件，只是一种更有条理、可共享、易发现的方式，用于记录这些命令行工具和脚本，并将其开放给Agent使用。

### 核心要点

SKILL是更合理的抽象设计。它将基于 “脚本编写” 的Agent模式正式化，比MCP那种僵化的类接口模式更稳健、更灵活。

## MCP（模型上下文协议）

SKILL的出现，并不意味着MCP就此过时（可参考我的《MCP的所有问题》一文）。此前，很多人搭建的MCP十分糟糕，不仅上下文臃肿，还包含数十个镜像 REST 接口的工具（比如`read_thing_a()`、`read_thing_b()`、`update_thing_c()`）。

如今被SKILL正式化的 “脚本编写” 模式虽更优，但需要一种安全的环境访问方式 —— 在我看来，这就是MCP新的、更聚焦的定位。

MCP不应再是臃肿的接口集合，而应成为一个简单、安全的网关，提供几个强大的高层级工具，比如：

- `download_raw_data(filters…)`（下载原始数据）
- `take_sensitive_gated_action(args…)`（执行受权限管控的敏感操作）
- `execute_code_in_environment_with_state(code…)`（在带状态的环境中执行代码）

在这个定位下，MCP的职责不再是为Agent抽象现实场景，而是管理身份验证、网络连接和安全边界，随后便不再介入后续工作。它只为Agent提供一个入口，Agent再通过脚本编写能力和 markdown 上下文完成实际工作。

我目前唯一还在使用的MCP，是与 Playwright 配套的那个 —— 这也合情合理，因为 Playwright 是一个复杂的有状态运行环境。所有无状态工具（如 Jira、AWS、GitHub），我都已迁移为简单的命令行工具。

### 核心要点

将MCP用作数据网关，为Agent提供一两个高层级工具（比如原始数据导出接口），让它在此基础上编写脚本开发。

## Claude Code 开发工具包（SDK）

Claude Code 不仅是一个交互式的命令行工具，还是一个功能强大的开发工具包，可用于搭建全新的Agent —— 无论是编码还是非编码任务。如今做新的副业项目时，我都会将它作为默认的Agent开发框架，而非 LangChain、CrewAI 这类工具。

我主要有三种使用方式：

1. 大规模并行脚本开发：做大规模代码重构、bug 修复或迁移时，我不会使用交互式聊天功能，而是编写简单的 bash 脚本，并行调用`claude -p “在/pathA目录下，将所有foo引用替换为bar”`这类指令。这种方式的可扩展性和可控性，远胜于让主Agent管理数十个子Agent任务。
2. 搭建内部聊天工具：这个开发工具包非常适合将复杂流程封装成简单的聊天界面，供非技术人员使用。比如一个安装程序，出错时会调用 Claude Code 开发工具包自动修复问题；又比如公司内部的 “本地快速原型” 工具，让设计团队能基于公司自研的 UI 框架，随心编写前端原型代码，确保设计想法的高保真度，同时让这些代码能更直接地用于前端生产环境。
3. Agent快速原型开发：这是我最常用的方式，而且不仅限于编码领域。只要有任何Agent相关的开发想法（比如一个使用自定义命令行工具或MCP的 “威胁调查Agent”），我都会先通过 Claude Code 开发工具包快速搭建并测试原型，再决定是否投入资源搭建完整的部署架构。

### 核心要点

Claude Code 开发工具包是一个强大的通用Agent开发框架。处理代码批处理、搭建内部工具、快速开发Agent原型时，都可以优先使用它，而非更复杂的框架。

## Claude Code GitHub Action（GHA）

Claude Code 的 GitHub Action 功能，或许是我最喜欢、也被大家严重低估的一个特性。它的设计理念很简单：在 GitHub Action 中运行 Claude Code，但正是这份简单造就了它的强大。

它和 Cursor 的后台Agent、Codex 的托管网页界面类似，但可定制化程度要高得多。你能完全控制整个容器和运行环境，从而获得更多的数据访问权限，更重要的是，它能提供比其他任何产品都更强的沙箱隔离和审计管控能力，还支持钩子函数、MCP等所有高级功能。

我们利用这个功能搭建了自定义的 “全场景代码合并请求” 工具：用户可以从 Slack、Jira，甚至 CloudWatch 告警中触发代码合并请求，GitHub Action 会自动修复 bug 或开发新功能，并最终返回一个经过完整测试的代码合并请求。

由于 GitHub Action 的日志就是Agent的完整运行日志，我们制定了公司级的运维流程，定期审核这些日志，找出常见错误、bash 指令问题和不符合工程规范的操作。这就形成了一个数据驱动的正向循环：发现 bug → 优化`CLAUDE.md`/ 命令行工具 → 让Agent的表现更出色。

示例指令：

plaintext

```
$ query-claude-gha-logs --since 5d | claude -p “看看其他ClaudeAgent都卡在了哪些问题上，修复后提交一个代码合并请求”
```

### 核心要点

GitHub Action 是将 Claude Code 工程化落地的终极方式。它能让 Claude Code 从个人工具，转变为公司工程体系中核心、可审计且能自我优化的组成部分。

## settings.json 配置文件

最后，我总结了几个对副业和工作都至关重要的`settings.json`配置项：

1. `HTTPS_PROXY`/`HTTP_PROXY`（代理配置）：调试时非常实用。通过代理可以监控原始网络请求，清晰看到 Claude 发送的所有提示词。对于后台Agent，这也是实现精细化网络沙箱隔离的强大工具。
2. `MCP_TOOL_TIMEOUT`/`BASH_MAX_TIMEOUT_MS`（超时配置）：我会调高这两个配置的数值。我常运行一些耗时久、复杂度高的命令，而默认的超时时间往往过于保守。虽然现在 bash 后台任务功能已经上线，可能不再需要这个配置，但我还是会保留，以防万一。
3. `ANTHROPIC_API_KEY`（API 密钥）：工作中，我们通过 apiKeyHelper 工具使用企业级 API 密钥。这让我们的计费模式从 “按席位” 变成了 “按使用量”，更适配我们的工作方式：
    
    - 能适配开发者之间巨大的使用量差异（我们发现工程师的使用量差距可达 1:100）；
    - 工程师可以随意调试非 Claude Code 的大语言模型脚本，所有调用都计入公司的单一企业账户。
    
4. `permissions`（权限配置）：我会定期自查允许 Claude 自动运行的命令列表，做好权限审计。

### 核心要点

`settings.json`是实现高级自定义配置的重要入口。
