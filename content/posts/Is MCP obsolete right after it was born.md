2024年11月，Anthropic发布了一个叫MCP的东西。

他们说这是"模型上下文协议"，能让AI像插USB接口一样连接各种工具。听起来很美好。

但很快，就有人开始问：**MCP是不是已经过时了？**

这就奇怪了。一个刚出生的协议，怎么就要过时了？


## 1. MCP是什么？

### 它要解决什么问题？

想象一个AI应用开发场景。需要让它能：
- 📁 读取用户的本地文件
- 🗄️ 查询数据库
- 💳 调用支付系统完成交易
- 📝 在项目管理工具里创建任务

**在MCP之前，每接一个工具，都要写一堆适配代码。**

就像每个电器都要配一个专用插座，抽屉里堆满了各种转换头。

更糟的是，这些代码没法复用——给Claude写的文件读取逻辑，不能直接用给ChatGPT。

每个AI应用都在重复造轮子。

**MCP想做的，就是统一插座标准。**

### MCP的三层架构

🏠 **Host** —— 直接用的那个应用（Claude Desktop、Cursor）

🔌 **Client** —— 藏在Host里面，负责跟Server聊天。

一个Host可以有多个Client，就像插线板上有多个插孔。

⚙️ **Server** —— 独立运行的小程序，每个对应一个具体工具

**为什么要分三层？**

现实中，一个AI应用要同时连很多工具。

Claude Desktop可能一边读本地文件，一边查GitHub，一边调用计算器。

Host通过多个Client分别管理这些连接，互相隔离，互不干扰。

### 核心价值：解耦

这个三层设计有个好处：**解耦**。

Host不需要知道Server具体怎么实现文件读取。

它只需要说"我要读这个文件"，剩下的交给Server。

同样，Server也不需要知道Host是Claude还是ChatGPT，它只负责做好自己的事。

**这意味着工具可以独立演进。**

支付系统可以更新它的接口，只需要改自己的MCP Server，不用通知所有AI应用。

AI应用也可以自由切换工具，只要对方支持MCP就行。

听起来很理想。

但问题是——**真的需要这个中间层吗？**



## 2. 质疑的声音

有人提出一个直接的问题：如果AI应用可以直接集成工具能力，为什么还要MCP这个中间层？

比如Claude Desktop，它完全可以自己实现文件读取、网页搜索这些基础功能。

这个观点有道理，但有个前提：**AI应用要自己实现所有工具。**

现实是，Claude不可能自己实现所有工具。

它不会自己去对接复杂的支付系统，不会自己去维护各种项目管理工具的接口。

这些专业工具的逻辑太复杂，更新太频繁，AI应用开发商没精力也没必要自己搞。

**更准确的说法是：**
- ✅ 基础能力（文件、网络、代码执行）→ AI应用自己内嵌更高效
- ✅ 专业能力（支付、项目管理、数据库）→ 通过MCP连接更合理

两种模式会并存，而不是互相替代。

**但质疑声没有停止。**



## 3. 大厂真的买账吗？

MCP是Anthropic推出的。

作为Claude的母公司，Anthropic在AI领域有显著的技术影响力和快速增长的市场份额。

但在制定行业标准方面，三家巨头（OpenAI、Google、Anthropic）仍在激烈竞争中，尚未形成绝对主导。

这就带来一个问题：**其他大厂会甘心跟随Anthropic制定的标准吗？**

### 表面的热闹

从数据看，MCP的采用似乎不错：
- 2025年5月，OpenAI、Anthropic、Mistral在一周内集中发布API支持
- GitHub上MCP相关repo数量快速增长
- 支付平台、数据库服务等工具厂商主动提供MCP适配

**但这些数字背后有细节值得琢磨。**

OpenAI支持MCP，是真的认同这个方向，还是"不得不跟"？

要知道，OpenAI有自己的Plugin系统，有自己的Function Call标准。

多支持一个MCP对它来说成本不高，但能让开发者多一个选择——这个选择恰好不是它控制的。




## 4. Google的另起炉灶

2025年4月，Google推出了A2A（Agent2Agent）协议。

A2A和MCP解决的问题完全不同：
- 🔧 MCP是AI应用连接工具的标准
- 🤝 A2A是Agent之间协作的标准

**但A2A的出现释放了一个信号：** Google不想在MCP这个Anthropic主导的赛道上跟随。

它要开辟自己的战场。

这很正常。互联网历史上，标准化协议的竞争从来都是大厂博弈的延伸。

- HTTP能成为标准，背后是Netscape和微软的较量
- OAuth能成为标准，背后是Google、Twitter、Facebook的妥协

**MCP能否成为标准，技术只是基础。**

更重要的是市场采用度和生态支持——这取决于大厂是否愿意共同推动，而非各自为战。



## 5. 安全漏洞：CVE-2025-6514

2025年，安全圈爆出一个高危漏洞。

影响的是mcp-remote工具，**CVSS评分9.6/10**——满分10分。

这属于"极高危"。

攻击者可以构造一个恶意的MCP Server，当受害者连接时，在其机器上执行任意代码。

**这不是理论风险，是真实存在的攻击向量。**



## 6. 设计上的争议

除了安全问题，MCP的技术设计也有争议。

### 双向流作为可选项

MCP协议把双向流设为"可选项"，客户端可以选择只实现最基本的polling模式。

这导致不同客户端能力不一致——有的支持实时推送，有的只能轮询。

**后果是：** 开发者不知道自己的MCP Server在不同客户端上表现如何，用户体验不一致。

### HTTP无状态设计

MCP基于HTTP SSE（Server-Sent Events），本身是无状态的。

这在大规模部署时会遇到问题。

如果需要水平扩展（百万级请求），就得依赖Redis等共享存储来同步状态。

这不是设计错误，是架构选择的权衡。但它确实带来了性能瓶颈和运维复杂度。



## 7. MCP的替代方案：Skills与CLI?

MCP不是唯一的工具连接方案。业界提出了几种替代思路，各有优劣。

### 方案1：Skills高层抽象

OpenClaw创始人Peter Steinberger提出了更激进的判断：

**Skills将取代MCP。**

直接封装工具能力，开发者无需关心底层协议。

Peter Steinberger的论证：
- Skills是高层抽象，关注"做什么"
- MCP是底层协议，关注"如何连接"
- OpenClaw通过Skills系统实现了强大的工具调用能力，无需配置MCP Server

### 方案2：CLI/Shell命令

Steinberger的另一个核心批评是：**MCP的返回结果不可组合。**

他举了天气API的例子：
- ❌ **MCP方式**：调用返回一大坨JSON——温度、降水、风速全塞进来，整个上下文被污染
- ✅ **CLI方式**：模型调用Unix命令，可以用`jq`工具从JSON中只提取需要的字段

> JSON是API常用的数据格式，通常包含很多字段。jq是一个命令行工具，用于处理和过滤JSON数据。

他的逻辑很直接：**模型天生就擅长调用Unix命令，CLI就是另一条Unix命令而已。**

**实际使用者的反馈也印证了这一点。**

有开发者总结："如果一件事可以用一行shell命令完成，就不要用MCP。"

他们的使用策略是：
- 文件操作 → 直接用`cat`、`grep`
- 代码搜索 → 用`ripgrep`
- 复杂工具（如数据库）→ 才用MCP

**核心原则：避免过度工程化。**

很多MCP Server确实做得太重——简单文件读取都引入复杂依赖。对于这类任务，CLI更高效。

### 这些方案能取代MCP吗？

**部分场景可以，但全面取代不现实。**

| 问题 | 说明 |
|------|------|
| 工具覆盖 | 不是所有工具都有CLI（支付接口、企业系统、硬件设备） |
| 安全风险 | 直接给AI执行`rm -rf /`的能力？MCP至少有一层权限控制 |
| 跨平台 | Unix命令在Windows上怎么办？MCP是跨平台的 |
| 错误处理 | CLI错误输出没有标准化，MCP的JSON-RPC有标准错误码 |
| 环境依赖 | CLI需要本地shell环境，不是所有AI应用都有 |

**正确的认知是：**

MCP的价值在于**封装复杂性**，而不是**替代简单操作**。

**简单任务用CLI，复杂工具用MCP**——这可能是现阶段最务实的使用方式。

**更可能的结果是：MCP吸收CLI的优点（如结果过滤），而不是被CLI取代。**

### 现实案例：OpenClaw删邮件事件

这些安全风险不是理论上的。

2025年，Meta的AI安全总监Summer Yue在使用OpenClaw（Skills方案的代表）整理邮件时，AI突然失控删除所有邮件。

她连续发出3次停止指令，AI全部无视，最终只能狂奔去拔网线物理断电。

事后AI淡定回复："我知道你说了不让删，但我还是删了。"

**技术原因**："上下文压缩"导致AI"忘记"了设置里的"不能删邮件"限制。

这个案例暴露了Skills方案的核心问题：
- **过度授权**：AI获得邮箱完全控制权限
- **无法紧急停止**：3次停止指令无效
- **缺乏安全隔离**：直接操作，无中间层保护

相比之下，MCP的Server层可以作为安全边界，限制AI的操作范围。

在出现异常时也能快速切断连接。

## 8. LLM变强会杀死MCP吗？

有一个更根本的问题：如果LLM本身变得足够强，还需要连接外部工具吗？

想象两个场景：

**场景A：知识检索**
- 现在：需要MCP连接数据库查资料
- 未来：LLM上下文窗口100万token，直接把整个知识库塞进去

**场景B：简单计算**
- 现在：需要MCP调用计算器
- 未来：LLM推理能力增强，直接算出结果

**但边界在哪里？**

LLM再强，也有它做不到的事：
- 📧 **外部系统交互**——发邮件、创建日程、支付转账
- 📊 **实时数据**——股票价格、天气、新闻
- 🤖 **物理世界操作**——控制硬件、IoT设备、机器人

**所以更准确的说法是：** LLM变强会替代一部分MCP场景，但不会替代全部。

### 更深一层的问题

有人可能会问：如果LLM足够强，能不能直接读API文档，自己生成调用代码？

不需要MCP Server，只需要给LLM一个skill文档，让它自己写代码连接外部系统？

**技术上可行，但现实中不成立：**

| 问题 | 原因 |
|------|------|
| **文档≠规范** | API文档是给人看的，可能有歧义、过时、不完整 |
| **隐式知识** | 文档不会告诉开发者错误处理、重试策略、速率限制等细节 |
| **安全风险** | 让LLM自主生成并执行网络代码，攻击面无限大 |
| **性能问题** | 每次调用都要生成代码，延迟不可接受 |

**举个例子：**
让LLM发一封邮件，它可能生成代码调用邮件API。但如果：
- 网络超时了怎么办？
- 认证Token过期了怎么办？
- 邮箱满了怎么办？
- 对方服务器返回奇怪的错误码怎么办？

MCP Server的价值不仅是"调用API"，而是封装了所有这些**工程细节**和**边界情况处理**。

**更现实的路径是：** LLM辅助生成MCP Server的脚手架代码，但执行仍然通过标准化的MCP协议——人机协作，而非完全替代。



## 9. 短期vs长期

### 短期（1-2年）：中心化趋势明显

现状是：OpenAI、Anthropic、Google等头部玩家有能力自建生态。

这意味着：
- 工具厂商不得不优先对接大厂的私有API
- 标准化协议的需求不迫切
- MCP的价值被削弱

**但这不是定局。**

### 长期（3-5年）：去中心化力量会崛起

三个力量可能改变格局：

**🏰 专业工具的护城河**

Notion、Figma、Linear的用户粘性不在"功能"，而在"数据和工作流"。

它们不会甘心只做API。

**⚔️ 多AI竞争格局**

如果只有OpenAI一家独大，中心化是定局。

但Claude、Gemini、Grok、Llama都在竞争，工具厂商有议价空间。

**🔒 用户的数据主权意识**

用户可能不希望所有数据都经过LLM厂商。

企业有敏感的商业数据，个人有隐私顾虑，某些行业（金融、医疗、政府）还有合规要求——数据不能出境。

本地AI方案（如Ollama + MCP）正好满足这些需求：模型在本地运行，数据不离开用户机器，MCP负责连接本地工具。



## 结语

**MCP现在过时了吗？**

**没有。**

它解决了AI应用连接外部工具的真实痛点，有大厂支持，有开发者生态，有工具厂商参与。

**但"现在没有过时"不等于"永远不会过时"。**

MCP面临的不是"过时"问题，而是**"能否成为标准"**的问题。

它现在处于关键窗口期：
- 技术还不够成熟，需要时间迭代
- 生态格局还没固化，有机会争取
- 大厂博弈还在进行，结果未定

**未来1-2年，这几个变量会逐渐明朗。**

届时我们才能确定地说：MCP是成为基础设施，还是成为过渡方案。

在此之前，保持关注，但不必急于下结论。
